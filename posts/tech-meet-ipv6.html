Transitioning to IPv6: Dual Stack Without Double Trouble

Reflection on an evening session about IPv6 adoption opportunities, risks, and practical steps to deploy IPv6 safely in real infrastructures.

The main message of this session was direct: the transition to IPv6 is not a “future topic”, it is an operational reality. ISPs and large organisations have already moved forward, and SMEs are increasingly forced to follow not because IPv4 suddenly stops working, but because the ecosystem around it is becoming more expensive, more complex, and less scalable.

What I appreciated most is that IPv6 was not presented as a simple upgrade. IPv6 changes how networks behave: addressing becomes abundant, NAT is no longer the default crutch, and the end-to-end model becomes realistic again. That sounds cleaner but it also means visibility and control must be rebuilt properly. “Security by design” here means: don’t treat IPv6 as a side feature; treat it as a first-class citizen in routing, firewalling, monitoring, and documentation.

The strongest practical takeaway was about dual stack. In many environments, the real transition phase is not IPv6-only it’s IPv4 + IPv6 running together. And dual stack can be dangerous if handled casually: you can unintentionally double your attack surface, create bypass paths, and end up with inconsistent rules (for example, a firewall policy that is strict on IPv4 but permissive on IPv6 “because nobody uses it yet”). That mindset is exactly how quiet exposures happen.

IPv6 also introduces protocol-specific risks teams must respect: router advertisements (RA), neighbour discovery (NDP), address auto-configuration (SLAAC), and the way devices select routes. These are powerful features but they require guardrails. The “secure by default” approach is to explicitly define what is allowed on IPv6, enable RA controls where needed, keep a default-deny posture, and make logging/monitoring IPv6-aware (so incidents don’t disappear into blind spots).

My personal action plan after this session is very concrete: start with inventory and visibility (which systems already have IPv6 enabled?), check DNS readiness (AAAA records, resolver policies), review firewall parity (IPv4 and IPv6 rules aligned), and validate tooling (monitoring, SIEM, vulnerability scanning, and incident response playbooks that support IPv6 properly). If an organisation treats IPv6 as “just more addresses”, it will become a security gap. If it treats IPv6 as a structured transition program, dual stack becomes a safe bridge instead of a hidden risk.
